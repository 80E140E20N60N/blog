JS执行机制

# JS执行机制

javascript是一门单线程语言，单线程的核心概念是指：同一个时间只能做一件事。即 JS任务 一个一个顺序执行。如果一个任务耗时过长，那么后一个任务必须等着。
单线程导致的问题是：如果某个 JS任务 执行的时间过长，会导致页面的渲染不连贯。

不管是什么新框架新语法糖实现的所谓异步，其实都是用同步的方法去模拟的。

# 事件循环Event Loop

事件循环是js实现异步的一种方法，也是js的执行机制。

# 同步任务和异步任务

为了解决 JS任务 阻塞的问题，将任务分为两类：

  • 同步任务
  • 异步任务

打开网站时，网页的渲染过程就是一大堆同步任务，比如页面骨架和页面元素的渲染。而像加载图片音乐之类占用资源大耗时久的任务，就是异步任务。

  • 同步和异步任务分别进入不同的执行"场所"，同步的进入主线程，异步的进入Event Table并注册函数
  • 当指定的事情完成时，Event Table会将这个函数移入Event Queue
  • 主线程内的任务执行完毕为空，会去Event Queue读取对应的函数，进入主线程执行
  • 上述过程会不断重复，也就是常说的Event Loop(事件循环)

怎么知道主线程执行栈为空？
js引擎存在monitoring process进程，会持续不断的检查主线程执行栈是否为空，一旦为空，就会去Event Queue那里检查是否有等待被调用的函数。

console.log('代码执行开始');
setTimeout(() => {
    console.log('延时3秒');
},3000)
console.log('代码执行结束');

# 宏任务和微任务

除了广义的同步任务和异步任务，我们对任务有更精细的定义：

  • macro-task(宏任务)：包括整体代码script，setTimeout，setInterval，UI rendering
  • micro-task(微任务)：Promise.then()或catch()，Node.js独有的process.nextTick

事件循环的顺序，决定js代码的执行顺序。event loop它的执行顺序如下：

  • 一开始整个脚本作为一个宏任务执行
  • 执行过程中同步代码直接执行，宏任务进入宏任务队列，微任务进入微任务队列
  • 当前宏任务执行完出队，检查微任务列表，有则依次执行，直到全部执行完
  • 执行浏览器UI线程的渲染工作
  • 检查是否有Web Worker任务，有则执行
  • 执行完本轮的宏任务，回到2，依此循环，直到宏任务和微任务队列都为空

注：在所有任务开始的时候，由于宏任务中包括了script，所以浏览器会先执行一个宏任务，在这个过程中你看到的延迟任务(例如setTimeout)将被放到下一轮宏任务中来执行。

setTimeout(function(){
    console.log('延时')
})

new Promise(function(resolve) {
    console.log('promise')
    resolve();
}).then(function() {
    console.log('then')
})

console.log('打印')

进阶分析：

console.log('1');       //立即执行

setTimeout(function() {    //宏任务队列，二轮事件循环时执行
    console.log('2');      // 二轮事件循环 立即执行
    process.nextTick(function() {
        console.log('3');  // 二轮事件循环 微任务
    })
    new Promise(function(resolve) {
        console.log('4');  // 二轮事件循环 立即执行
        resolve();
    }).then(function() {
        console.log('5')   // 二轮事件循环 微任务
    })
})
process.nextTick(function() {
    console.log('6');   //微任务
})
new Promise(function(resolve) {
    console.log('7');   //立即执行
    resolve();
}).then(function() {
    console.log('8')    //微任务
})

setTimeout(function(){      //宏任务队列，三轮事件循环时执行
    console.log('9');       // 三轮事件循环 立即执行
    process.nextTick(function() {
        console.log('10');  // 三轮事件循环 微任务
    })
    new Promise(function(resolve) {
        console.log('11');  // 三轮事件循环 立即执行
        resolve();
    }).then(function() {
        console.log('12')   // 三轮事件循环 微任务
    })
})

# setTimeout 和 setInterval

# setTimeout

setTimeout是异步的，可以延时执行，注意，这里的延时指的是从异步任务进入Event Table开始计时，计时结束，异步任务进入Event Queue。 如果主线程任务耗时较长，那么异步任务也需要等待主线程任务执行完毕后，才能执行。这就是为什么setTimeout的延时时间不准的原因，例如：

setTimeout(() => {
    console.log('延时');
},3000)

function fib(n){
    if(n==1 || n==2){
        return 1;
    }
    return fib(n-2)+fib(n-1)
}

let res=fib(46)
console.log(res);

# setInterval

setInterval和setTimeout差不多，不同点是setInterval是设置间隔，循环执行。

setInterval每隔指定时间将注册的函数置入Event Queue，如果主线程任务耗时较长，同样需要等待，对于setInterval来讲，同样存在计时不准的问题，即：
对于setInterval(fn,ms)来说，不是每过ms秒会执行一次fn，而是每过ms秒，会有fn进入Event Queue。一旦setInterval的回调函数fn执行时间超过了延迟时间ms，那么就完全看不出来有时间间隔了
